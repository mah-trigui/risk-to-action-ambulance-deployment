Your Model Predicts Risk. Operations Needs Actions.

Translating crash-risk predictions into ambulance deployment decisions in Nairobi

View Repository Explore Method

Overview

This project addresses a city-scale ambulance deployment problem in Nairobi. The task was to place 6 ambulances every 3 hours in locations that would minimize response distance to future traffic crashes.

The key design decision was to separate the system into two different layers:

  • prediction layer: estimate crash probability for each road segment and time window
  • decision layer: convert the resulting risk surface into 6 deployable ambulance positions

Instead of trying to predict coordinates directly, the system first estimated where crashes were likely, then translated those predictions into an operational plan.

Architecture

The Core Idea

A predictive model is not the same thing as a decision system.

The model output in this project was a probability:

P(crash | road segment, 3-hour window)

But the operational output needed by the system was completely different:

6 latitude/longitude positions for ambulances

The value came from building the translation layer between those two spaces: from risk estimation to constrained deployment.

System Decomposition

Stage 1 — Crash prediction

Nairobi was decomposed into road segments, and an analytical base table was created at the segment × 3-hour window level. A binary model then estimated the likelihood of at least one crash in each slot.

Stage 2 — Deployment translation

The highest-risk segments for each time window were selected, then clustered into 6 geographic groups so ambulance positions could be placed at cluster centroids.

Why this matters

Prediction produces a risk surface. Operations needs a small set of actions under resource constraints. Those are related, but they are not the same object.

Analytical Base Table Design

The city was transformed into a tabular prediction problem by building one row per:

  • road segment
  • 3-hour time window

This converted a spatial deployment problem into a structured binary classification task.

Features were then fused from multiple sources:

  • crash history: segment counts, seasonal medians, recent crash elasticity
  • road survey: 228 obfuscated attributes compressed with t-SNE
  • Uber Movement: segment-level speed aggregated into 3-hour windows
  • weather: humidity, temperature, wind, precipitable water
  • temporal context: weekends, holidays, hour groups
  • spatial context: geographic segment clusters

Why this project is interesting

  • shows how a predictive model can feed an operational decision layer
  • separates score-space from action-space
  • demonstrates how to convert large risk maps into small deployable plans
  • combines machine learning, spatial reasoning, and resource allocation

Where this pattern generalizes

  • fraud risk scoring vs investigation queue allocation
  • hospital risk prediction vs staffing assignment
  • demand forecasting vs inventory placement
  • outage prediction vs field crew dispatch

The recurring pattern is the same: the output of the model is not yet the output of the operating system.

Key Takeaway

Your model predicts risk. Operations needs actions.

Real-world ML systems become useful when a decision layer translates predictions into constrained operational choices.
